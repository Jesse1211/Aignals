# Aignals v0.1 — Master Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship Aignals v0.1 — a macOS menu bar app that flips a red/green dot in response to AI agent activity, driven by Claude Code hooks writing JSON files into `~/.aignals/sessions/`.

**Architecture:** Three loosely-coupled tiers. (1) `aignals-hook` shell CLI is the producer — Claude Code lifecycle hooks invoke it, it writes/updates/removes `~/.aignals/sessions/<session_id>.json`. (2) The filesystem is the protocol. (3) `Aignals.app` is a SwiftUI `MenuBarExtra` consumer that watches the directory via FSEvents and renders aggregate state.

**Tech Stack:** Swift 5.9 / SwiftUI / AppKit (Aignals.app, macOS 13+), bash + jq (aignals-hook CLI), Swift Package Manager (build + test), XCTest (Swift tests), bats-core (CLI tests), `xcodebuild` (CI builds), `create-dmg` (packaging), Homebrew Cask (distribution).

**Spec:** [`docs/superpowers/specs/2026-06-16-aignals-menubar-design.md`](../specs/2026-06-16-aignals-menubar-design.md). This plan implements that spec exactly. If a conflict arises, the spec wins — flag it and stop.

## Subagent parallelism map

Most phases serialise (later phases consume earlier types), but a few can run in parallel by independent subagents:

- **Phase 0** must finish first (scaffolding).
- **Phases 1, 2, 6** are independent of each other after Phase 0 → run in parallel.
- **Phase 3** needs Phase 2 (`Session`).
- **Phase 4 & 5** both depend on Phase 1 + 3 → can run in parallel after 3.
- **Phase 7 (E2E)** is the integration gate; needs 1–6 finished.
- **Phase 8** can start in parallel with 7 (consumes only `SessionStore`).
- **Phases 9 & 10** are independent of 7/8 → can run after Phase 8 in parallel.
- **Phase 11** is last.

```
Phase 0
  ├──> Phase 1 ────┐
  ├──> Phase 2 ──> Phase 3 ──┬──> Phase 4 ──┐
  └──> Phase 6                └──> Phase 5 ──┴──> Phase 7
                                                  └──> Phase 8 ──┬──> Phase 9
                                                                  └──> Phase 10
                                                                          └──> Phase 11
```

## Commit cadence

Each phase commits after each step that produces a working tree change. A typical step ends with:

```bash
git add <files> && git commit -m "<phase-prefix>: <one-line summary>"
```

Phase prefixes match the table above (`phase-01:`, `phase-02:`, ...). Use Conventional Commits — `feat`, `test`, `chore`, `docs`, `fix`, `build` — inside the phase prefix when natural, e.g. `phase-03: feat: add SessionStore.upsert`.
